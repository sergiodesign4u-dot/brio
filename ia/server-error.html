500 Server error

The generic server-error fallback: deliberately minimal and self-contained so it renders even when the backend and its data are unavailable. Only the Brio wordmark, a 500 mark, an honest non-blaming line, a data-is-safe reassurance, and a small static recovery (try again, home, status). It carries no app data, no dynamic includes, and no external assets, which is the deliberate contrast with the richer, context-aware 404. Returns a real HTTP 500 and is noindex. A failed check-in submit is handled by 10.6, not this page. Source of truth: ia/docs/pages/server-error.md.

Mobile-first priority

Content blocks

Minimal by design: the page must survive an outage, so it holds only what can render without the backend. The failure is conveyed in text, never by color.

  1. Minimal chrome: only the Brio wordmark, no full nav and no footer. Unlike the 404, the 500 stays minimal and static, because app chrome that needs data cannot be trusted when the backend is down.
  2. Error mark: a large "500" and H1 "Something went wrong", plus a plain, non-blaming line ("this is on our side, not you"). Text, not color.
  3. Reassurance: the account and the team's data are safe and nothing was lost (it matters for a wellbeing and privacy product), plus "we have been notified".
  4. Recovery (small, static): "Try again" (reload), "Go to homepage" (a static link), and a quiet "check status" link. No backend call is required to recover.

Components used: the 11.0 error-mark kit (wf-errcode, wf-errh, wf-errsub) and the Brio wordmark, plus a small static action row. No nav, footer, search, or app chrome, on purpose. No new global; a failed check-in submit is handled by 10.6 (answers preserved).

Black and white, wireframe level

Wireframe

No color and no brand: structure only, color arrives at Concept (stage 06). The same minimal, static page renders everywhere; there is no context-aware variant, because a server fault means app chrome cannot be relied on. Compare with the 404, which is richer and context-aware.

Technical and SEO

Technical and SEO (noindex)

A 500 is a system response, not indexable content, so there is no A to E SEO block; correctness and resilience are what matter.

Self-contained so it survives an outage. The page is fully self-contained, inline CSS, no external fonts, scripts, or images, and no dynamic includes or API calls, so it renders during a backend failure. Ideally it is served as a static fallback from the edge or CDN. This is why it holds no nav, footer, search, or app chrome.

  • HTTP status: the page must return a real 500 (the 404 returns 404, the 503 returns 503)
  • robots: noindex; no schema.org
  • reassurance is honest: it says the data is safe because the aggregate-only architecture means a request failure does not expose or lose individual data; it does not over-promise
  • the LCP is the heading text, not an image; the recovery links are static <a> and the retry is a reload
  • a failed check-in submit is handled by 10.6 (answers preserved), not this generic 500
A calm, honest fallback that always renders. 500 Server error is the minimal, backend-independent page: the Brio wordmark, a 500 mark, an honest non-blaming line, a data-is-safe reassurance, and a small static recovery, returning a real HTTP 500 and noindex. It reuses the 11.0 error-mark kit and invents nothing global; carrying no app data or external assets is its defining property and the deliberate contrast with the richer 404.
