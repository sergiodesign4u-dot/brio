Invite team

The final onboarding step: invite the team so the program has people to hear from, then finish into the Dashboard. Inviting is encouraged but not required to finish, so the operator is never blocked. Serves the main job and inviting the team; it closes the main flow F1. Inside the operator app, so noindex. Source of truth: ia/docs/pages/onboarding-invite.md.

Mobile-first priority

Content blocks

Block 1 is the first screen on mobile. Order reasoned from the 360px baseline up.

  1. Wizard header: the Brio mark and "Step 3 of 3".
  2. Heading: a warm final-step line ("Invite your team").
  3. Invite block: the join link with a copy control, the join code, and an email-invite field. The same in-node invite element as 6.0.
  4. Privacy note: the roster wall, condensed, "The people you invite are never linked to who answered" (0.3 condensed).
  5. Finish action: Finish and go to your dashboard (to 4.0), primary. Inviting is not required.
  6. Invite later: a secondary way to finish without inviting now (still to 4.0); the team can be invited from 6.0 any time.

Components used: 0.3 Privacy-disclosure (condensed), the in-node invite element shared with 6.0, and the wizard shell from 3.0. No new global invented.

Private app zone

SEO block (noindex)

Invite team is a transactional, authenticated wizard step. It is not indexed and carries no schema. The in-app findability rules still apply.

robotsnoindex, nofollow
canonicalnone for search; app URL is /app/onboarding/invite
schema.orgnone (transactional wizard)
hreflang / OGnone

In-app findability (not SEO, still specified)

  • exactly one H1 (the heading) for the accessible document outline
  • the join code and link are selectable text with labelled copy controls
  • the email field is a labelled input; the progress indicator states "Step 3 of 3" in text
  • tap targets at least 44px
