Log in

Return access. Log in is a supporting page, not an acquisition surface, but it gates every recurring job (Main, read the pulse, catch a decline early, feel competent), so it must be fast, forgiving, and never a dead-end. Email + password or magic link, with a forgot-password path. Reuses the auth-page shell from 2.0. Public but transactional, so the form is noindex. Source of truth: ia/docs/pages/login.md.

Mobile-first priority

Content blocks

Block 1 is the first screen on mobile. Order reasoned from the 360px baseline up. The chrome mirrors 2.0 so the two auth pages read as one system.

  1. Pre-auth header: the Brio mark and a single "Sign up" link (the mirror of 2.0's header), no job tabs.
  2. Heading: a short welcome-back line ("Log in to Brio").
  3. Log-in form: a labelled email field and a labelled password field, with "Forgot password?" by the password, primary CTA "Log in".
  4. Magic-link option: "Email me a link instead" (passwordless), to 2.2 Magic-link sent. Sidesteps a forgotten password entirely.
  5. Sign-up link: "New to Brio? Create an account", to 2.0, so a person on the wrong page is never stuck.
  6. Footer: 0.2 footer, public variant (condensed).

Components used: a reduced 0.1 (pre-auth header), 0.2 (public footer), and the auth-page shell from 2.0. No new global. The condensed 0.3 trust line from 2.0 is deliberately omitted (a returning operator needs speed, not re-selling). SSO is a paid feature, absent from standard log-in.

Public but functional, return access

SEO block (noindex, follow)

Log in is a public but purely functional return-access page. It carries no acquisition value and is not indexed, the same treatment as 2.0 and different from the deep app zones (noindex, nofollow).

robotsnoindex, follow (thin, transactional; follow keeps equity flowing to Sign up and legal)
nav queries"Brio login" is navigational, best served by Home / Landing which links here; the form need not rank
schema.orgnone on the form
hreflang / OGnone

In-app findability (still specified)

  • exactly one H1 (the heading) for the accessible document outline
  • email and password are labelled inputs with visible labels
  • the wrong-credentials message is conveyed in text, not by color alone
  • Forgot password, Log in, and the magic-link option are labelled controls; tap targets at least 44px
The recurring gate. Log in reuses the auth-page shell from 2.0, a reduced 0.1, and the 0.2 public footer; it invents nothing global. It is fast, forgiving (wrong credentials and forgotten passwords both have a clear path), and never a dead-end. Next: 2.2 Magic-link sent (the "check your email" state, shared by the magic-link and password-reset paths), then 2.3 Invalid or expired link, which closes cluster 2.
