Category / segment landing

The SEO acquisition template: one canonical structure that renders N indexable target pages, one per program category and one per company-size segment, each with a per-target hero, unique intro, program list, and value points, in the public chrome. It is not a single hand-built page, it is a template with token slots; the block structure and SEO shape are fixed, the per-target copy varies and must be genuinely unique so the pages are not thin or duplicate. The target list is a hypothesis pending keyword research. Serves the existing Main job (acquisition), an IA mechanism, not a new job. Indexed, with a full A to E SEO block. Source of truth: ia/docs/pages/segment-landing.md.

Mobile-first priority

Content blocks (fixed skeleton, token slots)

The structure is fixed so every target page is consistent and maintainable; the tokens in {braces} change per target. The block that must be genuinely unique per target is the intro, or the pages become thin and duplicate.

  1. Public nav (0.1 pre-auth): the Brio mark, marketing links, Log in, Start free. No job tabs. (Fixed.)
  2. Hero: H1 {target} plus a per-target subhead and Start free. The LCP is the heading text.
  3. Target intro {intro}: one or two paragraphs written for this category or segment, the block that must be unique per target (anti-thin).
  4. What is included {programs}: the relevant programs, a light list from the 0.4 and 0.6 catalog.
  5. Why Brio for {target}: two or three value points (private by design, guided, low effort), reworded per target.
  6. Privacy proof: the subscription-as-privacy line (0.3 claim 4, public), link to 1.1. Internal links: 1.0 Pricing, 1.1 Security, 0.0 Home, siblings. CTA band + Footer (0.2).

Components used: 0.1 Navigation (public), 0.2 Footer, the public chrome from 1.0 and 1.1, the subscription-as-privacy strip (0.3 claim 4), and a lightweight program list from the 0.4 and 0.6 catalog. The token map, program list, value points, and internal-links row are thin, page-specific layout. No new global; the target list and per-target copy are a hypothesis pending keyword research.

Exactly one H1 per target, then a fixed H2 skeleton.

C. Ready SEO text

Pattern: {Target} for a small company with no HR: Brio gives you a guided set of {category or segment} programs, runs light weekly check-ins, and shows an honest, aggregate-only team signal, never an individual answer.
Example (mental health): Mental health programs for small teams with no HR: Brio gives you a guided set of mental-wellbeing check-ins, runs them for you on a light weekly cadence, and shows an honest team signal that is aggregate-only, so no one ever sees an individual answer. Start free and run your first program in about five minutes.

D. Structured data (schema.org, per target)

  • BreadcrumbList: Home, {section}, {target}
  • CollectionPage or ItemList: the programs in the "what is included" block (must match the visible list)
  • Organization: name Brio, url, logo ? asset at Concept

E. Optimization checklist (the template's guardrails)

  • exactly one H1 per target; the LCP is the hero heading text, not an image
  • canonical is self per target, never a shared master, or the set will not index
  • each target has a genuinely unique title, description, intro, and program list; near-identical pages are the thin and duplicate failure mode of a landing template
  • only publish a target with real search demand and real unique copy; the target list is ? pending keyword research, not auto-generated at scale
  • siblings cross-link, and every page links to 1.0 Pricing, 1.1 Security, 0.0 Home, 2.0 Sign up; nav and footer links are crawlable <a>
  • canonical and hreflang set; Core Web Vitals budget respected; final query volumes deferred to production ?
