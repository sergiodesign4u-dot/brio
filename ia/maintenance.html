Maintenance (503)

The planned-downtime notice: a calm, expected message that Brio is briefly unavailable for scheduled maintenance and will be back, not an apologetic error. Only the Brio wordmark, a calm status pill and friendly heading, a plain reason, an optional expected-return chip, and a small static recovery. It is minimal and self-contained like the 500, but the tone is a notice, not a fault. It returns a real HTTP 503 with a Retry-After header (the defining difference) and is noindex. Source of truth: ia/docs/pages/maintenance.md.

Mobile-first priority

Content blocks

A maintenance window is a normal event, not a failure, so the tone is calm and there is no large error numeral. Like the 500 it is minimal and self-contained, because the app may be intentionally offline.

  1. Minimal chrome: only the Brio wordmark, no full nav and no footer, self-contained.
  2. Status and heading: a calm status pill ("Scheduled maintenance") and a friendly heading ("Brio is getting an upgrade" / "Back shortly"). No large error numeral; the 503 is a small technical detail, not a headline.
  3. Plain reason: "we are doing some scheduled maintenance and will be back shortly".
  4. Expected return (optional): an estimated time back ?, shown as a chip so the wait is bounded; omitted if unknown.
  5. Recovery (small, static): "Try again" (reload) and a quiet "check status" link. No account push, no upsell.

Components used: the 11.1 minimal card (wf-mincard) and the Brio wordmark, a status pill, an optional expected-return chip, and a static action row. No nav, footer, search, or app chrome. No new global; the calm-notice tone is what distinguishes it from the 500.

Technical and SEO

Technical and SEO (noindex)

A maintenance page is a system response, not indexable content, but the status code is the whole point and differs from the 404 and the 500.

Return HTTP 503 with a Retry-After header. The page must be served with a real 503 Service Unavailable and a Retry-After header (a number of seconds, or a date). This is the defining difference: 503 tells search engines and monitors the site is temporarily unavailable, not gone, and Retry-After tells them when to come back, so a planned maintenance is not de-indexed or penalized. Contrast: 404 means gone, 500 means an unexpected fault, 503 means back soon.
  • HTTP status: 503 plus Retry-After (the 404 returns 404, the 500 returns 500, neither carries Retry-After)
  • robots: noindex; no schema.org
  • self-contained: inline CSS, no external assets or dynamic includes, served from the edge or a static host so it renders while the app is offline
  • the LCP is the heading text, not an image; the retry is a reload and the status link is a static <a>
A calm, bounded wait, and search-engine-safe. Maintenance is the planned-downtime notice: the Brio wordmark, a status pill and friendly heading, a plain reason, an optional expected-return chip, and a small static recovery, returning a real HTTP 503 with a Retry-After header and noindex. It reuses the 11.1 minimal card and invents nothing global; its calm, expected tone and the 503 + Retry-After status are the deliberate contrast with the apologetic 500 and the gone-for-good 404. This completes the error-page trio. Next: 11.3 Cookie consent (grounded in real law), then 11.4 Toast, and the 11.5 Global search N/A confirmation.
