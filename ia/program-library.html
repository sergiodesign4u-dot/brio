Program library

The operator browses the curated library and picks a program to run, with the size-matched recommendation pinned so a non-expert is guided, not dropped into a catalog. This is the first listing, so it also sets the listing template. Serves Main, J6, and item 2 (self-serve start). Inside the operator app, so noindex. Source of truth: ia/docs/pages/program-library.md.

The planks a listing adds

Listing template

A listing adds three planks to the per-page template from Home (0.0). Program library is the first built to it; 5.1 and any later grid follow the same shape.

  1. Facet set defined in-node. Filter values live with the listing unless they are big enough to be a canonical global. Brio's set is thin (one dimension, category), so it is defined here and cross-referenced, not lifted into 0.6.
  2. The grid reuses a canonical card. The cells are 0.4 Program card. The listing arranges and filters cards; it never restyles them.
  3. Empty and loading are first-class states. Here that is 5.3 first-run empty (a registered node) and the 0.4 skeleton. Neither is ever a dead-end (MS4).
Mobile-first priority

Content blocks

Block 1 is the first screen on mobile. Order reasoned from the 360px baseline up.

  1. Nav + page header (0.1 operator, Programs tab active): H1 "Programs", one-line subhead "Pick a program. Brio runs it and reads the team pulse for you."
  2. Recommended-for-your-size card (0.4 recommended state, pinned): the guided shortcut that closes choice paralysis. The page's primary action.
  3. Facet bar (category): All plus the four categories (0.6). Thin, single dimension. See Facets below.
  4. Program-card grid (0.4 canonical): the rest of the library as default cards; programs the operator already runs carry status badges.
  5. Empty state (5.3, first run): shown in place of the grid before anything is picked; recommended shortcut plus browse hint. Never a dead-end.
  6. App shell footer (minimal): a privacy link (1.1) and a help link, not the public marketing footer (0.2).

Components used: 0.1 Navigation (operator logged-in, Programs active), 0.4 Program card (recommended / default / active / draft / ended / skeleton), 0.6 Canonical lists (categories, size buckets). No new component; the facet set is in-node, not a global.

Listing only

Filters and facets

One facet, defined here because the set is thin and used mainly on this page. It filters which 0.4 cards show; it never changes a card.

FacetValuesBehavior
Category (single-select)All (default), Mental health, Physical, Financial, Social (from 0.6)client-side narrowing over a small curated set; reflected as ?category= for the back button; canonical stays /app/programs
  • The recommended pin stays first only under All; within a filtered category the size-recommended program, if present, sorts first.
  • No search, no pagination, no sort control in the MVP. Browse plus one facet is enough for a small curated library, which is the concrete reason 11.5 Global search is likely N/A (confirmed or closed at Step 6). A future sort is deferred ?, not built.
  • Zero result in a category shows "Nothing here yet, see all programs" that clears the facet back to All. Never a dead-end.

Private app zone

SEO block (noindex)

Program library is inside the operator app, an authenticated per-account view. It is not indexed and carries no schema. The in-app findability rules still apply.

robotsnoindex, nofollow
canonicalnone for search; app URL is /app/programs, facet as ?category=
schema.orgnone (not a public landing page)
hreflang / OGnone

In-app findability (not SEO, still specified)

  • exactly one H1 ("Programs") for the accessible document outline
  • the facet controls are labeled controls, not color-only
  • every card is one in-app <a> to 5.1 with the title as its accessible name
  • the recommended tag and status badges are text, not color alone; tap targets at least 44px
  • if a public "programs overview" or a 1.5 category landing ever renders these cards, that page is indexable and gets its own A-E SEO and schema decision at its own node, not here
