Reset password

The screen reached by clicking the reset link in the password-reset email. It closes the forgot-password loop opened at 2.1: take a new password, set it, sign out other devices, and return to log in. A supporting page that gates every recurring job, so it is fast, forgiving, and secure. Reuses the 2.0 and 2.1 auth-page shell and the 2.2 and 2.3 confirmation layout; routes a stale link to the 2.3 twin. Tokened URL, so noindex and, unlike 2.0 and 2.1, nofollow. Source of truth: ia/docs/pages/reset-password.md.

Mobile-first priority

Content blocks

Block 1 is the first screen on mobile. Order reasoned from the 360px baseline up. The chrome mirrors 2.0, 2.1, and 2.3 so the auth pages read as one system.

  1. Pre-auth header: the Brio mark and a single "Log in" link (the mirror used on 2.3), no job tabs, so a person who lands here by mistake can always leave to 2.1.
  2. Heading: a short, direct line ("Set a new password").
  3. New-password form: a labelled "New password" field with a Show or Hide affordance, a labelled "Confirm password" field, a requirements hint (length first, NIST-aligned: at least 12 characters, checked against known-breached passwords, no forced composition or rotation), primary CTA "Set new password". Requirement state is by shape and text, never color.
  4. Security reassurance line: a plain note that setting a new password signs out other devices, so the reset is honest about its effect.
  5. Footer: 0.2 footer, public variant (condensed).

Components used: the auth-page shell from 2.0 and 2.1, the 2.2 and 2.3 confirmation layout (solid check) for the success state, and 2.3 itself for the stale-link path. The requirements checklist is thin, node-specific layout; no new global is invented. There is no magic-link or SSO option here; this page has one job.

Validation, recovery, and security

States and primary CTA

Forgiveness and security are the theme: a weak or mismatched password is corrected in place with the form still filled, and a stale link routes to the existing twin rather than dead-ending.

Tokened, transactional

SEO block (noindex, nofollow)

Reset password is a tokened page with no acquisition value and a secret in the URL, so it is not indexed and, unlike 2.0 and 2.1, not followed either. This is the stricter treatment, chosen precisely because the URL carries a single-use secret token.

robotsnoindex, nofollow (the URL carries a single-use secret token; nothing to index, no equity to pass)
referrerno-referrer so the token is never leaked to any outbound resource
schema.orgnone on the form
hreflang / OGnone

In-app findability (still specified)

  • exactly one H1 (the heading) for the accessible document outline
  • New password and Confirm password are labelled inputs with visible labels
  • the requirements, mismatch, and weak-password messages are conveyed in text, not by color alone
  • Show or Hide, Set new password, and Log in are labelled controls; tap targets at least 44px
The closing half of the loop. Reset password reuses the 2.0 and 2.1 auth-page shell, the 2.2 and 2.3 confirmation layout, and 2.3 for stale links; it invents nothing global. It is secure (a single-use token, other devices signed out), forgiving (weak and mismatched passwords stay inline, never a dead-end), and honest (it says what it does).
