Dialog / modal shell

One accessible modal contract, defined once, reused by every dialog in the product: resend invite (6.1), upgrade (7.2), cookie consent (11.3), and destructive confirmations. Each instance fills the body and actions; all inherit the same chrome and the same focus and keyboard behavior. No H1. Source of truth: ia/docs/pages/dialog.md.

Behavior

States

  • Default (open): header, body, footer.
  • Loading: an action is in progress (payment, invite send); the primary button shows a busy state and is disabled, the dialog stays open.
  • Error: the action failed; an inline message appears in the body, actions return to enabled (for example 7.3 Payment failed inside 7.2).
  • Success: a brief confirmation, then the dialog closes and returns focus.

Grounded in WAI-ARIA APG

Accessibility contract

  • role="dialog" and aria-modal="true" on the container.
  • Labelled by the visible title with aria-labelledby (or aria-label when there is no title); aria-describedby only for a simple single-message body.
  • On open: focus moves to the first focusable element inside (or the container). For a destructive confirm, focus goes to Cancel.
  • Focus trap: Tab and Shift+Tab cycle within the dialog; focus cannot leave without closing.
  • On close: focus returns to the element that opened the dialog.
  • Escape closes; a backdrop click closes too, unless a variant blocks accidental loss (an in-progress payment).
  • Background is inert and visually dimmed; no interaction reaches it.
  • SEO: dialogs open on interaction and are not indexable; the shell is an interaction and a11y contract, not an SEO surface.

Source: WAI-ARIA Authoring Practices, Dialog (Modal) pattern, w3.org/WAI/ARIA/apg.

Canonical reuse. Defined once here, reused by 6.1, 7.2, 11.3, and every confirmation. Change the contract here and it holds everywhere. Next global component: 0.6 Canonical lists, then the pages begin (Home, 0.0).
