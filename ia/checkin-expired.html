Invite link expired

The employee-flow twin of 2.3: a clicked check-in link that is expired (the cycle closed, or the link aged out) or invalid (broken or mistyped) gets a plain, non-blaming explanation, a reassurance that nothing personal is exposed, and a recovery, never a wall. The employee cannot mint a new token, so the recovery is asking the operator to resend, plus an optional fresh join code. For Yemi the employee. Serves the F2 no-dead-end rule. Private token-gated surface, so noindex. Source of truth: ia/docs/pages/checkin-expired.md.

Mobile-first priority

Content blocks

Even the error keeps the calm near-anonymous surface, with the anonymity anchor still present. The error reads as different from success through the dashed "!" mark and words, not color (the 2.3 treatment).

  1. Minimal top bar with the anonymity anchor: the Brio mark, the company name if known, and the "Anonymous" reminder.
  2. Attention mark: a dashed badge with an "!" (not a check), so the error reads without relying on color.
  3. Plain, non-blaming explanation: "This check-in link has expired" or "we could not open this link", with a one-line reason. Adaptive for expired vs invalid.
  4. Reassurance: nothing personal is exposed and nothing was recorded; the privacy promise is unaffected.
  5. Recovery (never a dead-end): "Ask your operator to resend" is primary (the employee cannot self-serve a token); an optional "have a fresh code?" field continues to 10.1; a quiet "who is my operator?" hint.

Components used: the 10.0 check-in chrome (no nav, no account), the 2.3 attention treatment (dashed "!" badge), and the 10.0 join-code field for the optional recovery. No new global; no personal data appears, only the anchor and the reassurance line.

Black and white, wireframe level

Wireframe

No color and no brand: structure only, color arrives at Concept (stage 06). The error is drawn with a dashed "!" badge (distinct from the 10.3 solid check) and words, never color. Two variants are shown (expired and invalid), then the desktop card. The recovery is real: ask the operator, or enter a fresh code.

Behavior

States and primary CTA

StateWhat showsAction
Expired link"this check-in link has expired", the cycle closed or link aged outask operator to resend; optional code to 10.1
Invalid link"we could not open this link", broken or mistypedsame recovery
Join-code recoverya fresh code fieldvalid to 10.1; invalid shows inline (not a new dead-end)

Primary CTA: ask the operator to resend (guidance, since the employee cannot self-serve a token), with the optional join-code Continue as the in-screen recovery to 10.1. Unlike the auth twin 2.3, there is no "request a new link" button, because the employee does not control invites. No account step, no other navigation, by design.

Private token-gated surface

SEO block (noindex)

Invite link expired is a private, token-gated employee surface. It is not indexed and carries no schema. It is a transactional zone, per the platform SEO stance.

robotsnoindex, nofollow (token-gated, personal)
canonicalnone for search; flow URL is /c/{invite-token}
schema.orgnone
hreflang / OGnone

In-app findability (not SEO, still specified)

  • exactly one H1 (the explanation heading) for the accessible document outline
  • the error is conveyed by text and the "!" mark, never by color alone
  • the join-code field is a labelled input and Continue is a labelled control
  • tap targets at least 44px; the flow works without an account
An error, not a wall. Invite link expired is the employee-flow version of the no-dead-end rule that 2.3 applies in auth: a stale or broken link becomes a plain explanation plus a real recovery (ask the operator to resend, or enter a fresh code), never a wall. It reuses the 10.0 chrome, the 2.3 attention treatment, and the 10.0 join-code field, and invents nothing global; no personal data is shown.
