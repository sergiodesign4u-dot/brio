Invite-send error

The error surface behind a failed email send in onboarding step 3. It names which invites did not go out, lets the operator fix and retry, and always keeps a way forward: the join link and code are unaffected, so a bad email can never block onboarding. It inherits 3.2's job coverage and closes cluster 3. Inside the operator app, so noindex. Source of truth: ia/docs/pages/invite-send-error.md.

Mobile-first priority

Content blocks

Block 1 is the first thing the operator reads on mobile. Order reasoned from the 360px baseline up. This is a state of wizard step 3, not a new screen, so the wizard shell and "Step 3 of 3" stay.

  1. Error summary: a plain, non-alarming line stating how many invites did not send ("2 of 3 invites did not send"). No blame, no jargon.
  2. Failed list: the specific addresses that failed, each with a short reason hint and a per-row retry. A typo can be seen and fixed in place.
  3. Retry action: Retry failed (retries all still-failing rows). The primary corrective action.
  4. Reassurance (link and code still work): the join link and code shown again with copy and a one-line note that they are unaffected. The guarantee onboarding is never blocked.
  5. Skip for now: a clearly visible way to dismiss the error and return to 3.2, where Finish still completes onboarding to 4.0. Failed people can be invited later from 6.0.

Components used: the in-node invite element from 6.0 / 3.2 (link and code, here as the still-works reassurance), the wizard shell from 3.0, and an in-node inline-error panel. No new global invented; the error panel stays in-node until a second consumer justifies promoting it.

Retry, skip, and role

States and primary CTA

The never-a-dead-end rule is concrete here: Skip for now is available in every state, so a bad address can never trap the operator (main flow F1, milestone MS3).

Private app zone

SEO block (noindex)

Invite-send error is a transactional, authenticated wizard state. It is not indexed and carries no schema. The in-app findability rules still apply.

robotsnoindex, nofollow
canonicalnone for search; app URL is /app/onboarding/invite (error surfaced inline)
schema.orgnone (transactional wizard state)
hreflang / OGnone

In-app findability (not SEO, still specified)

  • exactly one H1 (the error heading) for the accessible document outline
  • the failed addresses and the join code and link are selectable text with labelled controls
  • the error is conveyed in text and a labelled tag, not by color alone
  • each retry and copy control is a labelled button; tap targets at least 44px
Cluster 3 closed. Invite-send error reuses the in-node invite element from 6.0 and the wizard shell; the inline-error panel stays in-node. It guarantees main flow F1 never dead-ends at MS3: a failed email cannot block onboarding, because the link and code always work and Skip for now always finishes.
