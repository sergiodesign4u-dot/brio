404 Not found

The canonical recovery page: a context-aware full page (public chrome for a guest, operator app chrome for a logged-in in-app 404) with a large 404 mark, a site and help search, quick links to the main destinations, and a contact fallback, so a wrong or dead URL is a recovery, not a wall. It returns a real HTTP 404 status (never a soft-404) and is noindex. It is the site-wide expression of the no-dead-end rule and confirms the 11.5 Global search decision (the search here is help and site search, not an app catalog index). Opens Step 6 (system nodes). Source of truth: ia/docs/pages/not-found.md.

Mobile-first priority

Content blocks

Recovery is the job, so the strongest elements are the search and the quick links. The failure is conveyed in text (a 404 numeral and a plain heading), never by color.

  1. Chrome (context-aware): the public nav and footer (0.1 / 0.2) for a guest, or the 7.0 app bar and job tabs (no tab active) for an in-app 404. The recovery content is identical; only the frame changes.
  2. Error mark: a large "404" and H1 "Page not found", with a plain, non-blaming line. Text, not color.
  3. Search: a search over Brio's public and help content (routes to 1.4 Help). This is help and site search, not an app catalog search (Brio has no catalog index, 11.5 is N/A).
  4. Quick links: Home, Help, Pricing, Security, and "your dashboard" (4.0) for a logged-in user.
  5. Contact fallback: "still cannot find it? contact us", a link to 1.3, so even a failed search is not a dead-end. Footer (0.2) or the app frame.

Components used: 0.1 Navigation and 0.2 Footer (public), or the 7.0 operator app chrome (in-app variant), the 1.4 help search (wf-search), the quick-links row, and the 1.3 contact fallback (wf-help). No new global; a bad employee check-in link is handled by 10.4, not this generic 404.

Technical and SEO

Technical and SEO (noindex)

A 404 is a system response, not indexable content, so there is no A to E SEO block; what matters is that the status code is correct.

Return a real HTTP 404, never a soft-404. The page must be served with a 404 status code. A "not found" page served with a 200 (a soft-404) is a defect: search engines would index the error and users would be misled. The custom 404 is configured at the host or CDN so any unmatched URL renders this page with a 404 status.

  • HTTP status: 404 for not-found; the sibling error pages 11.1 and 11.2 return 500 and 503 respectively
  • robots: noindex, follow (do not index the error; follow lets the recovery links pass and aids crawl recovery)
  • schema.org: none; error pages carry no structured data
  • nav, footer, and quick links are crawlable <a>; the LCP is the heading text, not an image
  • the search is help and site search (routes to 1.4), not an app catalog index; this confirms the 11.5 Global search N/A decision
A wrong URL becomes a recovery, not a wall. 404 Not found is the canonical recovery page: a context-aware full page (public or in-app chrome) with a 404 mark, a help and site search, quick links, and a contact fallback, returning a real HTTP 404 and noindex. It reuses the public chrome, the 7.0 app chrome, the 1.4 help search, and the 1.3 contact fallback, and invents nothing global. Next: 11.1 500 Server error and 11.2 Maintenance (503), the sibling error pages, then 11.3 Cookie consent, 11.4 Toast, and the 11.5 Global search N/A confirmation.
