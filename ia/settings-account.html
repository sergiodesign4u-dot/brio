Account settings

The operator's identity and access section: name, email, password and magic link, sign out, and delete account (GDPR erasure). It is supporting infrastructure, but it carries the one destructive action in the app, so delete is gated by the 0.5 confirm dialog and never the visual primary. Closes cluster 8. For Priya the operator. Authenticated app view, so noindex. Source of truth: ia/docs/pages/settings-account.md.

Mobile-first priority

Content blocks

Plain identity and access, with the destructive delete kept separate and carefully gated. Individual employee responses were never stored identifiably (aggregate only), which the delete confirm notes.

  1. Name: an editable field with Save.
  2. Email: the current email with a change-email action; a changed email is verified (reuses the 2.2 check-your-email pattern), so a typo does not lock the operator out.
  3. Password and sign-in: a change-password action, plus a magic-link toggle for passwordless sign-in (ties to the 2.x auth choices).
  4. Sign out: signs out to 2.1 Log in.
  5. Delete account (danger): a separated destructive action that opens the 0.5 confirm dialog; the confirm states what is erased and requires an explicit confirmation. GDPR erasure.

Components used: the 8.0 shell chrome, the auth input field (wf-input, from the 2.x pages), and the canonical 0.5 Dialog / modal shell for the delete confirm (same shell as 5.1 End program, 6.1 revoke, 7.2 upgrade). 8.4 is a new 0.5 consumer. No new global is invented.

Primary CTA: per field, Save; for access, Change password and the magic-link toggle; Sign out to 2.1. The delete-account action is deliberately separated, destructive, gated by the 0.5 confirm dialog, and never the visual primary; Cancel is the safe default.

Authenticated app view

SEO block (noindex)

Account is inside the operator app, an authenticated per-account view. It is not indexed and carries no schema.

robotsnoindex, nofollow (authenticated app)
canonicalnone for search; app URL is /app/settings/account
schema.orgnone
hreflang / OGnone

In-app findability (not SEO, still specified)

  • exactly one H1 ("Account" within the settings outline) for the accessible document outline
  • every field is a labelled input; the magic-link toggle state is by position and text, not color
  • the delete action and its confirm are labelled; the destructive step reads by an attention mark and words, not color
  • the confirm dialog is role=dialog with aria-modal, a focus trap, Esc to close, and focus returned to the opener; tap targets at least 44px
Careful with the one destructive action. Account handles identity and access and the single destructive step, delete account (GDPR erasure), gated by the 0.5 confirm that states what is erased and requires an explicit confirmation, with Cancel as the safe default. It reuses the shell chrome, the auth input, and the 0.5 dialog, and invents nothing global. This closes cluster 8.
